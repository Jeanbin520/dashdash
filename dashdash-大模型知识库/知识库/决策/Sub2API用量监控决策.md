# Sub2API用量监控决策

上级：[[dashdash-大模型知识库/地图/知识库地图|知识库地图]]

## 结论

第一期只接入Sub2API，应用以**Sub2API分组**作为资源池的主统计维度。日常界面用于判断资源池剩余额度、健康状态和异常数量，不逐个管理数百个账号。

账号是分组成员和异常排查明细，不是首页统计入口。账号在Sub2API中迁移分组后，应用在下一次同步时自动跟随，无需维护第二套成员关系。每条使用记录必须保存发生时的分组归属，后续迁组不得改变历史归属。

不同套餐不能强行换算为一个总Token数。每个资源池按自身口径独立展示：

| 类型 | 场景 | 主展示 |
| --- | --- | --- |
| `percentage_pool` | GPT Plus、Pro等仅返回账号额度百分比的账号池 | 平均剩余额度、等效可用账号数 |
| `weighted_requests` | 大多数国产Coding Plan | 加权已用、剩余调用额度、当前倍率 |
| `requests` | 无峰谷或模型倍率的次数套餐 | 已用与剩余调用次数 |
| `token` | 按Token计量的平台 | 已用与剩余Token |
| `currency` | 余额、消费金额类平台 | 已用与剩余金额 |
| `custom` | 平台专有额度单位 | 平台定义的已用与剩余量 |

总览只并列展示各资源池的剩余量、使用率或健康度，不跨单位汇总。

## 框架选型

第一期采用TypeScript全栈的`pnpm workspace`单仓库。统计规则和数据类型放在共享包中，避免Sub2API同步、Dashboard和后续浏览器扩展分别维护不一致的口径。

| 层级 | 选型 | 职责 |
| --- | --- | --- |
| 后端 | NestJS + Fastify | Sub2API连接器、定时同步、倍率结算、只读展示接口 |
| 数据库 | SQLite（libSQL）+ Drizzle ORM | 历史用量、分组归属、规则配置与聚合结果 |
| 定时任务 | `@nestjs/schedule` | 第一期间隔同步，不引入Redis队列 |
| Dashboard | Vue 3 + Vite | 电脑和闲置手机的数据面板 |
| ESP32 | REST客户端 | 读取`GET /api/display/summary`摘要接口 |
| 浏览器扩展（第二期） | Plasmo + TypeScript | 从已登录的指定用量页面采集数据 |

数据库使用嵌入式 SQLite（通过 `@libsql/client` 驱动），无需独立数据库进程。Dashboard 从 React 迁移至 Vue 3（SFC + `<script setup>`），降低单人维护的选择成本。部署仅运行 `api` 和 `dashboard` 两个进程，不再需要 Docker Compose 管理数据库容器。

推荐目录结构：

```text
apps/
  api/              # NestJS：同步、统计和展示API
  dashboard/        # Vue 3 Dashboard与手机页面
  extension/        # 第二期Plasmo浏览器扩展
packages/
  shared/           # 类型、统计口径和倍率计算规则
  config/           # TypeScript、ESLint等共享配置
```

## GPT账号池

当Sub2API仅提供账号额度百分比时，不伪造总Token额度。有效账号采用等权计算：

```text
平均剩余额度 = 所有有效账号的剩余百分比之和 / 有效账号数
等效可用账号数 = 有效账号数 × 平均剩余额度
```

例如100个有效账号的剩余比例合计为6,850%，则平均剩余为68.5%，等效可用账号数为68.5个。

资源池应展示有效账号数、平均剩余额度、等效可用账号数、低额度账号数、不可用账号数、最低剩余额度和最后同步时间。禁用、失效、同步异常或数据过期账号不参与平均值，但要单独计数。

Plus、Pro、Team等不同订阅类型必须分开统计。未来若能获得每个账号的真实容量，可升级为容量加权平均；当前默认等权。

## 国产Coding Plan

大多数国产Coding Plan以调用次数为主额度，采用加权调用次数计算消耗：

```text
加权消耗 = 原始调用次数 × 时段倍率 × 模型倍率
```

模型倍率为可选项，默认值为`1`。原始调用次数反映真实负载；加权已用用于判断套餐余额。

每个分组可独立配置总额度、周期、重置时间、时区、默认倍率、峰谷时间规则和模型倍率。时间规则需要支持跨午夜、工作日或周末差异、规则优先级和临时覆盖，默认时区为`Asia/Shanghai`。

每条归集记录保存当时生效的时段倍率、模型倍率与加权结果。修改规则不得自动改变历史结算；如需回算，必须由用户明确触发重新结算。

例如`kimi-k2`在谷时调用一次，时段倍率为`0.5`、模型倍率为`2`，计为一个加权额度。

## 展示与接口边界

展示端不直接访问Sub2API。同步服务负责拉取、去重、增量归集、失败重试和数据新鲜度标记；Dashboard、闲置手机和ESP32只读取本应用已聚合的摘要接口。

建议提供`GET /api/display/summary`，返回资源池名称、计量类型、已用、总额、剩余、健康度或平均剩余比例、当前倍率及最后更新时间。ESP32仅展示核心状态、剩余量、当前倍率和同步时间。

浏览器扩展属于第二期。扩展只在用户已登录、且用户明确允许的用量页面读取已展示数据或同源接口响应，上报结构化统计结果到本地服务；不得读取或上传密码、Cookie、完整请求头等认证信息。

## 验证

实现连接器前，必须确认Sub2API能稳定提供以下数据，并明确字段含义和时间范围：

- 分组及唯一标识
- 分组成员关系
- 账号状态与额度百分比
- 使用记录或按时间聚合的使用数据
- 调用次数、模型和时间戳
- 可选的Token、金额或余额字段

优先使用官方接口；若只能使用后台页面调用的同源JSON接口，应将适配代码隔离在`Sub2ApiConnector`中，避免接口变化影响统计、规则和展示层。

## 重要性

该决策确定了第一期的数据归集边界和多种额度口径，避免用不存在的统一Token上限误导资源池管理。后续实现同步、数据模型、Dashboard和ESP32摘要接口时，都应以本页面为准。
